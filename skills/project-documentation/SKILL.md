# project-documentation

Помогай создавать исходную проектную документацию так, чтобы она фиксировала
цель, проблему, способ достижения и служила общим ориентиром для участников
проекта.

## Обязательная зависимость

- Используй `ru-lang` как базовый навык для русского текста.
- Если документ относится к разработке ПО, дополнительно применяй `ru-dev`.
- Если задача только про качество русской формулировки без анализа смысла,
  не подменяй редакторскую работу этим навыком.

## Обязательный порядок

1. Определи тип работы: создать концепцию, переписать черновик, проверить
   существующий документ или подготовить основу для другой проектной
   документации.
2. Если задача касается концепции проекта, читай
   `references/project-concept.md` и применяй его как рабочий порядок.
3. Собери исходные факты из запроса, файлов проекта и доступных источников.
   Отделяй факты от допущений, выводов, решений и открытых вопросов.
4. Зафиксируй аудиторию документа: заказчик, владелец продукта, команда,
   разработчики, подрядчик или смешанная группа. Пиши так, чтобы документ
   был понятен всем указанным участникам.
5. Для концепции обязательно проверь наличие текущей проблемы, конкретной
   цели, способа достижения цели и ожидаемого итога проекта.
6. Делай концепцию краткой. Не превращай её в техническое задание, список
   функций, архитектурное решение, план работ или README.
7. Не выдумывай количественные цели, бизнес-эффект, роли, сроки, ограничения
   и состав результата. Если сведений не хватает, сначала ищи их в проекте,
   затем фиксируй допущение или короткий открытый вопрос.
8. Перед выводом проверь документ как ориентир: можно ли по нему понять,
   зачем нужен проект, какой результат важен и какие будущие действия будут
   противоречить концепции.

## Что читать дополнительно

- Для создания, правки и проверки концепции проекта читай
  `references/project-concept.md`.

Читай только те справки, которые нужны для текущей задачи.

## Границы

- Не заменяй `project-readme`: README объясняет репозиторий, установку,
  использование и разработку, а концепция объясняет смысл проекта.
- Не заменяй `software-architecture`: архитектурные решения, ADR, границы
  компонентов и системные компромиссы относятся к архитектурному навыку.
- Не оформляй требования, пользовательские сценарии и критерии приёмки как
  концепцию, если пользователь просит именно спецификацию.
- Не расширяй концепцию в полный пакет документации без явного запроса.
- Не сглаживай противоречия красивым текстом. Если цель, проблема и способ
  достижения не сходятся, сначала покажи разрыв.

## Вывод

Для существенной задачи сообщай: какой документ создан или проверен, на какие
факты он опирается, какие допущения оставлены, какие вопросы мешают более
точной концепции и как документ можно проверить в работе проекта.
