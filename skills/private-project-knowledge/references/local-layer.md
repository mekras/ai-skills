# Локальный слой знаний проекта

Используй эту справку, когда нужно создать, подключить или проверить приватные
правила и знания проекта для агента.

## Что можно хранить

Подходящие сведения:

- локальные соглашения, которые не нужны всем участникам проекта;
- сведения о рабочем окружении конкретного пользователя;
- приватные ограничения клиента или команды без паролей и ключей;
- заметки о типичных ошибках агента в этом проекте;
- локальные правила запуска, проверки и отчётности;
- ссылки на внутренние ресурсы, если сам факт ссылки допустим в локальном
  файле.

Не храни в таких файлах:

- токены, пароли, приватные ключи, cookie, одноразовые коды;
- полные персональные данные, если задачу можно решить ссылкой на безопасное
  хранилище;
- сведения, публикация которых создаёт юридический или физический риск;
- данные, которые должны ротироваться, аудитироваться или иметь разграничение
  доступа.

Для настоящих секретов используй менеджер секретов, переменные окружения,
зашифрованное хранилище или штатный механизм платформы.

## Имена и расположение

Нейтральные варианты:

- `rules.local/`
- `.local/rules/`
- `.agent.local/`
- `notes.local/`
- `AGENTS.local.md`

Выбор зависит от угрозы:

- если нельзя раскрывать даже наличие слоя, не добавляй имя в tracked‑файлы;
- если достаточно не хранить содержимое, можно использовать tracked
  `.gitignore` и tracked инструкцию с упоминанием локального файла;
- если проект использует несколько агентов, лучше нейтральная директория с
  отдельными файлами по инструментам.

## Исключение из VCS

Для Git, когда наличие слоя тоже не должно попадать в репозиторий:

```bash
printf '%s\n' 'rules.local/' 'AGENTS.local.md' >> .git/info/exclude
```

Если раскрыть паттерн допустимо, можно добавить путь в tracked `.gitignore`.
Но это уже сообщает всем читателям репозитория, что такой локальный слой
предусмотрен.

Для других VCS используй локальный механизм исключений, который не коммитится.
Если такого механизма нет, прямо назови риск.

## Подключение к агенту

Перед подключением проверь, как текущий агент загружает правила:

- автоматически читает только tracked `AGENTS.md`;
- умеет читать дополнительные локальные файлы;
- позволяет указать файл правил в пользовательской конфигурации;
- может получить локальные правила только через явный запрос пользователя.

Если единственный механизм — tracked `AGENTS.md`, строгая скрытность
невозможна: ссылка на локальный файл сама станет следом в VCS. В этом случае
предложи один из вариантов:

1. Не подключать автоматически и читать локальные правила вручную по запросу.
2. Использовать пользовательскую конфигурацию инструмента вне репозитория.
3. Согласовать раскрытие самого паттерна и добавить в tracked `AGENTS.md`
   нейтральную опциональную инструкцию без содержимого приватных правил.

## Проверка перед завершением

Для Git проверь:

```bash
git status --short --ignored
git ls-files --error-unmatch rules.local 2>/dev/null
git diff --cached --name-only
```

Ожидаемый результат:

- приватные файлы не входят в `git ls-files`;
- приватные файлы не попали в staged‑изменения;
- если они видны в `--ignored`, причина исключения понятна;
- tracked‑файлы не раскрывают приватный путь, если это было требованием.

## Формат локального файла

Держи локальный файл коротким и проверяемым:

```markdown
# Локальные правила проекта

## Применять всегда

- ...

## Применять только при ...

- ...

## Не хранить здесь

- Секреты, токены, пароли и приватные ключи.
```

Не дублируй обычные правила проекта. Если правило безопасно и нужно всем,
лучше перенести его в tracked `AGENTS.md`.
