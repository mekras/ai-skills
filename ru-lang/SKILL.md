# ru-lang

Применяй русский язык аккуратно во всех русскоязычных текстах: сохраняй
смысл, проверяемость и естественный порядок слов.

## Обязательная зависимость

- Используй отдельный навык `ru-text` из
  `https://github.com/talkstream/ru-text` как основу редакторской проверки
  любых русскоязычных текстов.
- Не копируй и не пересказывай правила `ru-text` внутри этого навыка.
  `ru-lang` отвечает только за применение русского языка в прикладном и
  техническом контексте.
- Если текущая среда явно показывает список навыков, проверь, что `ru-text`
  доступен как самостоятельный навык.
- Если `ru-text` недоступен, продолжай работу по правилам `ru-lang`.
  Упоминай отсутствие `ru-text` только когда задача требует строгой
  редакторской проверки, установки навыков или диагностики качества навыка.
- Если правила расходятся, применяй их в таком порядке: явное требование
  пользователя, терминология проекта или предметной области, смысл исходного
  материала, затем рекомендации `ru-text` и `ru-lang`.

## Область ответственности

- Этот навык не заменяет правила проекта, редакционную политику, юридические
  требования или отраслевые ограничения.
- Используй `ru-lang` всегда, когда нужно сформулировать русский текст.

## Общие правила

- Используй букву «ё».
- Для русских кавычек используй только «…».
- Не используй английские типографские кавычки `“` и `”`, а также не заменяй
  ими русские кавычки `«` и `»`.
- Перед отправкой русского текста быстро проверь, что в нём не осталось
  символов `“` или `”`.
- Перед отправкой любого русского текста, включая промежуточные комментарии,
  быстро проверь, что в нём нет слов и словосочетаний из списков ниже, если
  для них не нужно явно обоснованное исключение.

## Термины и технические элементы

- По умолчанию используй слова русского происхождения, если это не искажает
  смысл, не ухудшает точность и не противоречит другим правилам.
- Не русифицируй идентификаторы, имена API, команды, параметры, названия
  пакетов, классов, методов и файлов.
- Избегай лишних англицизмов в обычном русском тексте, когда есть точный
  русский вариант: `ошибка`, `журнал`, `конфигурация`, `обновление`,
  `возможность`, `повторная попытка`.
- Используй «Список нежелательных слов» для проверки выбора слов. Слова из
  этого списка по умолчанию не используй. Если для конкретного контекста
  нужна точность, устоявшийся термин или иное обоснованное исключение, можешь
  оставить исходную форму. В остальных случаях используй подходящую замену,
  указанную после `→`.
- Проверяй словосочетания, написанные через дефис: если форма выглядит как
  калька с английского порядка слов, замени её на естественную русскую
  конструкцию. Для проверки используй «Список недопустимых словосочетаний».
  Вместо недопустимой формы используй подходящую замену после `→`.
- Если термин уже закреплён в коде, документации, интерфейсе или рабочем
  процессе, используй его последовательно, даже если он не безупречен с
  редакторской точки зрения.
- Для `URL` избегай тавтологии: по возможности пиши просто `URL`, а если нужно
  родовое слово — `адрес ресурса` или `веб-адрес` по контексту.
- Списки ниже применяй к создаваемому русскому тексту. Не изменяй цитаты,
  URL, команды, имена файлов, веток, пакетов, классов, методов, параметров и
  другие технические идентификаторы без явной задачи редактуры.

## Список нежелательных слов

- `practically` → `на практике`, `на деле`, `почти`, `по сути`
- `апдейт` → `обновление`
- `батч` → `пакет`, `набор`, `пакетный запуск`
- `конфиг` → `конфигурация`
- `лог` → `журнал`
- `пайплайн` → `сборочная линия`, `конвейер`
- `ревью` → `обзор`, `проверка`
- `ретрай` → `повторная попытка`
- `фича` → `возможность`
- `чек-лист` → `проверочный список`

## Список недопустимых словосочетаний

- `<…>-файл` → `файл <…>`
- `API-ключ` → `ключ API`
- `Git-репозиторий` → `репозиторий Git`
- `HTTP-клиент` → `клиент HTTP`
- `OAuth-приложение` → `приложение OAuth`
- `OAuth-токен` → `токен OAuth`
- `Obsidian-модуль` → `модуль Obsidian`
- `TLS-сертификат` → `сертификат TLS`
- `VPN-<…>` → `<…> VPN`
- `ИИ-модель` → `модель ИИ`
- `код-ревью` → `обзор кода`
