# project-rules-audit

Проверяй правила проекта как рабочую систему управления агентом: они должны
помогать агенту действовать точнее, не конфликтовать между собой и не тратить
контекст без пользы.

## Обязательный порядок

1. Уточни цель проверки, только если без этого нельзя безопасно продолжить:
   найти проблемы, предложить правки, внести изменения, сравнить текущее
   состояние с предыдущим или выдать полный отчёт.
2. Найди источники правил. Начни с `AGENTS.md` и известных файлов инструкций
   для агентов, затем ищи локальные правила по словам `agent`, `AI`, `assistant`,
   `ИИ`, `instructions`, `rules`, `prompt`, `policy`, `guidelines`. Если
   проект использует навыки, учитывай локальные, глобальные и подключённые
   символическими ссылками навыки, доступные агенту или явно упомянутые в
   проекте.
3. Построй карту правил: файл, область действия, адресат, приоритет,
   повторяемые темы, обязательность, срок жизни и связь с инструментами.
4. Проверь проблемы: дубли, противоречия, неоднозначности, непроверяемые
   требования, устаревшие сведения, слишком широкие запреты, разрастание,
   смешение проектных и переносимых правил, локальные копии глобальных правил
   или правил из символических ссылок, лишнюю нагрузку на контекст.
5. Самостоятельно перепроверь выводы: для каждого существенного замечания
   найди подтверждение в файлах или явно пометь его как предположение.
6. По умолчанию разбирай найденные проблемы по одной: сначала собери
   достаточно контекста для ранжирования, затем покажи только одну наиболее
   важную проблему, варианты решения и рекомендацию. После этого остановись и
   жди решения пользователя; после одобрения, отклонения или уточнения по
   текущей проблеме сразу покажи следующую проблему с вариантами решения и
   рекомендацией. Не спрашивай отдельно, продолжать ли разбор, пока не
   закончились найденные проблемы или пользователь не попросил остановиться.
   После отработки последней найденной проблемы явно сообщи, что все найденные
   проблемы отработаны. Полный список проблем выдавай только по явной просьбе
   пользователя.
7. Если пользователь сразу просит исправить правила, вноси минимальные правки,
   сохраняя смысл, приоритеты и локальные соглашения проекта.
8. После правок проверь результат: перечитай изменённые места, убедись, что
   новые формулировки не создают новых конфликтов, и назови остаточные риски.

## Что читать дополнительно

- Для полного проверочного списка и формата отчёта читай
  `references/audit-checklist.md`.

## Границы

- Не превращай аудит в переписывание всех правил с нуля, если цель —
  диагностика.
- Не удаляй правило только потому, что оно повторяется: сначала проверь,
  не задаёт ли повтор приоритет, область действия или важное усиление.
- Удаляй локальный дубль глобального или подключённого через символическую
  ссылку навыка только после проверки, что внешний источник реально доступен и
  локальная копия не добавляет проектной области действия.
- Не объединяй проектные правила с переносимыми без явного основания.
- Не считай найденный конфликт доказанным, пока не указаны обе конфликтующие
  формулировки или их точные места.
- Не добавляй новые правила «для полноты», если они не закрывают найденный
  риск или реальную ошибку агента.

## Вывод

По умолчанию не выдавай полный список проблем. Покажи одну наиболее важную
проблему, предложи варианты решения и дождись явного решения пользователя.
После решения по текущей проблеме сразу переходи к следующей. Если
последняя найденная проблема отработана, явно сообщи, что все найденные
проблемы отработаны. Если пользователь явно просит полный отчёт, сначала
перечисляй проблемы и риски, затем предлагаемые правки, после этого краткое
резюме. Для каждой проблемы указывай файл и строку, если они доступны, тип
проблемы, влияние на работу агента и конкретное действие.
